# 企业命题适配评分规则

## 目标

将参赛者的企业赋能方向与官网 108 个命题逐一匹配，先执行硬门槛，再对通过门槛的命题进行分层。最终推荐同时考虑思想匹配度与单人开发可交付性，避免把概念相似误判为可落地。

## 已确认边界

- 团队规模为 2-3 人，但核心开发能力按“参赛者个人 + Codex”计算。
- 队友可以承担行业研究、业务流程梳理、样本整理、测试和展示，不作为额外研发产能计算。
- 前台客户体验可以是结果，但内部流程、组织、数据和系统协同必须构成方案主体，才能列为高度适配。
- 企业 Agent 的可信定位是“业务执行与协作层”；ERP、SAP、CRM 等首先保留为权威记录系统。比赛方案不直接宣称一次性替换财务、库存等核心账务系统。

## 硬门槛

出现以下任一情况，原则上列为 `C / 不适配`：

1. 核心难点依赖具身智能、硬件、嵌入式、传感器、控制算法或专业计算机视觉。
2. 核心判断依赖医药、临床、材料、化工、金融风控等高专业或高责任领域知识。
3. 主要成果只是内容生成、品牌传播、消费者聊天或一次性分析，且没有企业内部行动闭环。
4. 无法在没有企业真实生产接口和敏感数据的情况下构造可信验证。

## 分层定义

- `A / 高度适配`：核心问题是企业内部流程、组织或数据系统重塑；Agent/Harness 和飞书人机协作是必要组成；单人开发可以完成代表性闭环。
- `B / 可改造`：可以使用 Agent，但核心仍偏营销、内容、消费者产品、单点工具，或范围过宽、行业依赖较重。
- `C / 不适配`：触发硬门槛，或与企业管理系统重塑方向明显不一致。

## Top 候选评分（100 分）

| 维度 | 权重 | 判断问题 |
| --- | ---: | --- |
| 企业管理/FDE 愿景匹配 | 25 | 是否以真实企业流程和业务结果为中心，而不是以模型能力为中心？ |
| 系统级重塑深度 | 20 | 是否跨岗位、跨部门、跨系统，并包含异常与责任交接？ |
| Agent/Harness 必要性 | 15 | 是否需要持续的观察、计划、执行、验证和恢复，而不是一次问答？ |
| 飞书人机协作适配 | 10 | 是否自然需要组织、消息、文档、任务、审批、表格或人工责任门禁？ |
| 单人开发可交付性 | 15 | 能否用模拟数据和公开接口做出代表性 E2E，而不依赖专业算法？ |
| 命题清晰度与可验收性 | 15 | 官网是否提供角色、痛点、现状、目标、指标或明确目标性产出？ |

## 证据等级

- `verified`：来自官网运行时 `topics` 数据、报名表或公开 Q&A，且本地已提取核验。
- `partial`：公开页面只显示部分内容，或需要登录/企业后续提供资料。
- `inferred`：由命题描述推导出的产品架构、指标或 Demo 方案，不属于企业官方要求。
- `unknown`：当前公开材料没有提供，不能用推测补全。

## 评分使用原则

1. 先执行硬门槛，再评分；高专业风险不能靠 Agent 关键词抵消。
2. 官网标题、详情正文和目标性产出优先于企业宣传语。
3. 只对通过硬门槛的 Top 候选给出数字分，108 题完整地图使用 A/B/C，避免伪精确。
4. 最终推荐必须同时给出首选、稳健备选和放弃其他高分题的原因。
