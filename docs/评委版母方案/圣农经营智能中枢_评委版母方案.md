## 1. 执行摘要

### 1.1 方案目标

圣农已经建成 SAP S/4HANA 及多套专业系统。公开资料显示，SAP 已与 SRM、CRM、WMS、智慧农场等 18 个外围系统进行数据互联。这些建设解决了交易、核算和专业业务的信息化问题，也为进一步开展经营分析提供了基础。

企业命题关心的问题更进一步：当销售、价格、库存、生产或市场发生变化时，管理者能否及时看到，能否快速查明原因，能否形成可审批、可执行、可检查的解决方案。

本方案不重做 ERP，也不另建一套业务账。方案以现有系统为事实来源，以经营问题为组织对象，将分散的数据、流程和责任串联起来，形成“发现问题、查明原因、辅助决策、责任执行、结果检查、管理改进”的工作机制。

### 1.2 方案回答的七个问题

| 管理问题 | 方案回答 |
|---|---|
| 同一指标在不同系统中含义不一致 | 统一 SKU、规格、区域、到手价、销量、库存和毛利等经营口径，并保留版本和生效时间 |
| 异常发现得晚 | 利用源系统状态和企业批准的判断条件，及时登记需要调查的经营信号 |
| 一个小问题无法判断影响范围 | 从单店、区域、跨部门到集团战略逐步扩大调查，每次只补充当前判断所必需的证据 |
| 决策材料只有结论，没有依据 | 形成包含事实、原因假设、证据缺口、备选方案、影响和风险的决策材料 |
| 决策与执行脱节 | 管理层批准后拆成具体任务，明确负责人、配合人、完成时间、前置条件和验收依据 |
| 任务刚下达就被误判为未完成 | 以 SAP、WMS、TMS、POS 等系统的正式状态为进度依据，到达相应业务节点后再检查执行质量 |
| 同类问题反复出现 | 每十个同领域、已关闭且证据完整的事件形成一次专项复盘，查找制度、流程、组织、系统、培训和风险管理方面的重复性短板 |

### 1.3 人工责任边界

本方案坚持“系统辅助、人工决策”。智能助手可以查找资料、归纳原因、比较方案、生成任务草案和跟踪进度；程序可以做对账、计算、规则检查和状态监测。但凡涉及价格、存栏、产能、市场进入、制度修订和组织调整，都必须经有权管理者批准。

### 1.4 研究边界与验证说明

> 受比赛周期和验证条件限制，课题组目前无法进入圣农的飞书租户，也未获得应用权限、接口凭证和真实业务样本。因此，本文已核实飞书公开文档和公开案例中明确展示的平台能力；对于圣农内部的实际权限、数据字段、流程配置和系统联动效果，本文只提出设计方案，不将其写成已经实现的事实。项目启动后，这些内容需通过现场访谈、权限确认、接口联调和代表性端到端测试逐项确认。

为避免把设计方案误写成企业现状，本文统一使用三种表述：

| 类型 | 含义 | 使用要求 |
|---|---|---|
| 公开事实 | 来自年报、公告、公司官方材料或平台官方案例 | 只引用原文明确支持的结论，不延伸未披露的原因和效果 |
| 方案设计 | 课题组根据企业命题提出的建议 | 说明前置条件、实施方法和验收方式 |
| 待企业确认 | 公开资料不足以判断的内部事实 | 列入访谈、样本和测试清单，不自行假定 |

## 2. 对企业命题的理解

### 2.1 “18 个系统已打通”与题目并不矛盾

公开资料所说的“系统打通”，主要说明数据可以在 SAP 和外围业务系统之间流转。这是企业信息化的重要基础，但还不等同于一个经营问题能够被快速发现和处理。

以价格异常为例，形成可靠判断至少要弄清：是否为同一 SKU 和规格，是页面标价还是消费者实付价，优惠由平台、门店还是厂商承担，是否有已批准活动，降价后销量、毛利和库存如何变化。这些信息可能分布在价盘、POS、促销审批、财务结算、库存、巡店记录和飞书沟通中。

题目中的“数据不打通”可以进一步理解为以下几个层次的差距：

~~~text
数据能够流转
→ 同一经营概念的口径一致
→ 分散证据能够组成完整问题
→ 决策能够经审批转化为责任任务
→ 新的业务数据能够检查执行结果
~~~

### 2.2 现有基础和待确认能力

| 内容 | 公开资料支持的判断 | 本方案的处理 |
|---|---|---|
| SAP S/4HANA 及外围系统 | 已建成重要的交易、核算和经营分析基础 | 继续作为正式事实源，不以新系统替代 |
| 飞书品质巡检实践 | 2024 年飞书官方案例披露，圣农集团曾使用多维表格、机器人和任务实现“发现、整改、专业复核、关闭” | 先确认当前是否仍在运行及可复用范围，再决定如何扩展到价格、库存和经营决策 |
| MOM/MES 或生产指挥能力 | 公开资料不足以证明具体产品、覆盖范围和当前状态 | 现场确认后优先复用，不在未调研的情况下重复建设 |
| POS、经销商和门店实际销售数据 | 公开资料无法证明订单级实付价、优惠承担方和门店库存可全面获取 | 列为价格与动销试点的首要数据条件 |
| Aily 智能助手 | 飞书官方材料展示了通用产品能力，不代表圣农已开通或可直接使用 | 在目标租户完成权限、工具、流程和代表性端到端测试后再纳入正式实施 |

### 2.3 公开数据提示的经营课题

公开数据可用于确定调查方向，不能直接用于给内部单位定责。本方案重点关注三类课题：

1. **口径变化与经营判断**：SAP 上线前后可能存在成本核算口径变化；月度简报中也可能存在抵消前、未经审计等口径。系统在做同比、勾稽和预测前，必须识别这些差异。
2. **渠道变化与价格管理**：直销、经销和出口渠道的销量、毛利和库存可能出现不同变化。如果缺少经销商和门店真实动销数据，仅看公司出货量无法准确判断市场需求。
3. **库存规模与产销协同**：2025 年末存货账面价值约 32.86 亿元，存货跌价准备为关键审计事项。深加工肉制品库存量和销量均有增长，同时受并表口径影响。这些数据说明库存、需求和产能需要联动分析，不说明企业已经发生系统性滞销。

## 3. 总体方案

### 3.1 一个基础、两个方向、一套执行机制

~~~mermaid
flowchart LR
    A["基层产生经营事实"] --> B["发现异常或经营机会"]
    H["管理层提出战略问题"] --> C["统一口径、证据和业务规则"]
    B --> C
    C --> D["调查原因并比较多种方案"]
    D --> E{"管理层审批"}
    E -->|"批准"| F["任务分解、责任到人、按业务节点执行"]
    E -->|"修改或驳回"| D
    F --> G["新数据检查执行结果"]
    G --> C
~~~

自下而上的路径从一个价格、一个门店或一次客诉开始，逐步判断是单点问题、区域问题，还是会影响产销协同和集团目标的问题。自上而下的路径由管理者提出市场、产能、存栏或海外业务问题，再调取企业内外部信息进行研究。两个方向最后都要经过管理层审批、任务执行和结果检查。

### 3.2 从一个数据点到一份决策材料

源系统首先只需提供最小必要信息。例如价格异常可以由“门店、SKU、实际价格、参考价格、时间、来源”几个字段触发。程序负责确认数据是否完整、价格是否超出企业设定的比较范围，并登记一条待调查信号。它不在这个阶段判定违规，也不预先判断事件严重程度。

智能助手接到信号后，先按单店范围查询必要信息。如果单店数据已能说明是合法促销或价签错误，就不继续扩大范围。如果无法解释，再查询同区域门店、动销、库存、客诉和竞品信息。只有新证据显示影响已跨越部门或触及集团目标时，才继续扩展调查。

最终形成的决策材料至少包括：

- 已确认事实、信息来源和时间；
- 仍然缺少的信息及其影响；
- 可能原因、支持证据、反例和其他解释；
- 可选方案、适用范围、预算和前置条件；
- 对销量、毛利、库存、现金流、产能和风险的影响范围；
- 停止条件、恢复办法和需要承担的风险；
- 建议的负责人、配合人和跟踪节点。

### 3.3 信息缺失时不强行下结论

调查中常会遇到无权限、接口不通、数据未报送，或现场情况只有人员掌握的问题。系统一方面继续查询已授权信息，另一方面根据数据责任目录，通过飞书向相应人员发送补充信息请求。

决策页面应明确展示：哪些信息已经取得，哪些暂时无法取得，请求已发给谁，对方是否已读，预计何时回复。如果缺失信息会改变方案选择，系统只能标明“证据不足”，不得自行补齐。

### 3.4 决策批准后如何推动执行

管理层批准的内容应包括决策版本、适用对象、预算、时间、停止条件和恢复办法。系统确认审批有效后，将决策拆成销售、库存、生产、养殖、财务和海外业务等任务。每项任务明确负责人、配合人、完成标准、前置任务、截止时间和延误后的升级路径。

当任务需要两个以上主要责任角色，或需要跨部门、跨法人协作时，系统可在完成权限校验后自动建立飞书事件协同群。群内只纳入当前阶段必须协作的人员，并同步已批准的决策、任务、证据、进度和截止时间。单人即可完成的任务只发任务卡，不额外建群。外部经销商、客户和供应商不进入内部事件群；确有需要时使用另行批准的外部协作方式。

### 3.5 执行检查要尊重业务进度

系统不能在任务下达后立即判定“未完成”。以门店铺货为例，应先读取业务系统中的正式状态：

~~~text
SAP 出库或发运
→ TMS 签收或到货
→ 门店收货和登记
→ 商品上架或 POS 首次销售
→ 价签和实际成交价检查
~~~

货物仍在运输或门店仍在登记时，系统只监测进度，不报“未上架”或“未标价”。只有相应业务状态已经到达，并超过企业认可的处理时间后，才进行质量检查。

### 3.6 从处理个案到改进管理

单个事件关闭后，系统保留事件背景、原因、证据、处置方案和最终结果。同一业务领域内，每十个已关闭、结果已验证且证据完整的事件形成一个复盘批次。重复打开或属于同一根本问题的事件不重复计数。

复盘用于比较问题背景、原因、解决过程和结果，判断是否存在重复性的制度、流程、组织、系统、培训或风险管理问题。智能助手只生成候选判断和改进选项；管理层审议后，才能形成正式改进任务。后续批次继续形成，并用同口径数据检查问题是否减少。

## 4. 三个经营案例

### 4.1 案例一：29.9 元参考价与 24.9 元门店价

> 本案例用于说明方案运行方式，不代表公开资料已证实圣农存在违规低价。

某门店同一商品的实际价格为 24.9 元，企业当期有效参考价为 29.9 元。两者相差 5 元只能触发调查，不能直接判定门店违规。系统需查明是否为同一规格，24.9 元是页面价还是消费者实付价，是否使用平台券、会员券或厂商活动，优惠由谁承担，是否有正式审批。

| 阶段 | 主要工作 | 形成结果 |
|---|---|---|
| 信号登记 | 记录门店、SKU、规格、时间、实际价、参考价、来源和判断条件版本 | 一条待调查价格信号 |
| 单店核实 | 查询价盘、实付价、优惠组成、促销审批、价签和 POS 记录 | 合法活动、规格差异、价签错误、未取得审批等备选解释 |
| 区域调查 | 查询同区域价格、销量、库存、客诉和竞品促销 | 判断是单店问题还是区域性变化 |
| 方案比较 | 比较定向优惠、规格区隔、临期处置、不跟价等方案 | 销量、毛利、库存、预算和价格外溢风险的比较 |
| 管理审批 | 确认适用门店、SKU、数量、时间、预算、停止条件和恢复办法 | 正式决策版本 |
| 任务执行 | 拆成销售、门店、库存和财务任务，需要多方协作时建立事件协同群 | 负责人、截止时间和验收依据 |
| 结果检查 | 在到货、登记、上架等节点到达后，检查价格、销量、库存和毛利 | 关闭、继续观察、升级调查或重新审批 |

如果竞争对手只让利三天就能抢量，系统需要在小时级完成核心调查，不能等月报出来后再反应。快速不等于简单跟价。管理者至少应看到三种选择：限定门店和数量的短期响应，通过规格区隔避免全渠道价格下沉，或在对手优惠难以持续时保持价格稳定。系统负责测算代价，管理层负责选择企业愿意承担的风险。

### 4.2 案例二：多区域降价和滞销如何传导到生产与存栏

如果六个销售区域同时出现降价、库存上升和动销放缓，这是需要进入集团分析的重要信号，但还不能直接判定全国白羽鸡需求已经下降。管理层需要先比较多种原因：全国或区域需求下降、竞争对手短期抢量、自身价格或规格不适合、经销商库存较高、区域之间相互影响，以及数据迟到或口径差异。

调查要同时读取门店实际销售、渠道库存、公司库存、退货、回款、毛利、促销和竞品信息，再与生产计划、冻品库容、养殖存栏和可调整周期联系起来。

| 时间尺度 | 重点判断 | 可选措施 |
|---|---|---|
| 近期 | 哪些区域和 SKU 最急，库存是否快速累积，价格是否超出承受范围 | 跨区域调拨、定向促销、临期处置、暂停补货、收紧生产排程 |
| 中期 | 需求变化是否持续，产品和渠道结构是否需要调整 | 调整产品结构、产能利用率、采购、冻品储备和渠道政策 |
| 长期 | 国内需求、出口机会、产能布局和养殖规模是否需要改变 | 调整存栏和产能计划，开拓海外市场，优化产品和基地布局 |

存栏和产能调整不能由系统自动执行。决策材料应同时呈现需求预测范围、饲养和屠宰周期、现有存栏、合同和采购约束、产能上下限、冻品库容、现金流影响和动物福利要求，由管理层选择方案。

### 4.3 案例三：管理者主动发起海外市场研究

当管理者提出“是否应进入某个海外市场”时，系统先明确目标、时间范围、资金上限、风险偏好和不能突破的约束。然后同时调查企业内部产能、成本、产品、认证、历史出口、客户和回款情况，以及外部市场规模、进口政策、检疫要求、关税、汇率、物流、竞争和政策风险。

外部信息必须记录来源、发布日期、适用市场和产品、许可条件和可比性。网络搜索结果只能用于发现线索；法规、准入和重大投资判断必须回到官方来源，并经法务、质量和业务负责人确认。

决策材料至少比较“不进入、小规模试单、与本地伙伴合作、逐步建立市场”等不同路径。管理层批准后，再分解为市场准入、认证、客户开发、产品改造、产能安排、物流、保险、交付和回款任务。试单结果再反过来校正市场判断。

## 5. 五组核心能力

原有 13 个模块按管理用途归为五组。每组都回答六个问题：为什么需要、依赖哪些信息、如何运行、形成什么结果、由谁负责、还需向企业核实什么。

### 5.1 第一组：数据与证据

| 模块 | 管理目的 | 主要产出 |
|---|---|---|
| M01 数据来源与采集 | 明确信息从哪个系统、哪个法人和哪张业务单据取得 | 数据源目录、接入方式、刷新频率、负责人和权限 |
| M02 可靠传输与原始记录 | 防止数据丢失、重复、乱序或无法追溯 | 原始记录、接收时间、质量状态、重试和异常记录 |
| M03 经营口径与证据管理 | 让不同系统对同一经营概念使用一致且可解释的口径 | 经营概念字典、指标口径、版本、数据对应关系和证据索引 |
| M04 经营信号检测与数据质量 | 区分真实经营异常和数据本身异常 | 待调查信号、判断条件版本、合法例外和数据质量告警 |

#### 为什么需要

这组能力解决“数据从哪来、是否完整、含义是否一致、能否回到原始凭证”四个基础问题。如果这些问题没有解决，后续分析只会更快地放大错误。

#### 信息来源和接入方式

| 来源 | 主要内容 | 优先接入方式 | 关键限制 |
|---|---|---|---|
| SAP/ERP | 订单、发货、采购、生产、库存、成本、财务和主数据 | 官方接口、业务事件、企业已有集成 | 不影响正式交易；优先只读 |
| WMS/TMS | 出入库、在途、签收、批次、库龄和异常状态 | 接口、状态回调、企业集成平台 | 对齐订单、批次和物流单号 |
| CRM/SRM | 客户、渠道、合同、询价、供应商和政策 | 接口或已有数据交换 | 商务敏感字段按岗位控制 |
| POS/零售平台 | 门店、SKU、实付价、优惠、退款、销量和门店库存 | 授权接口、消息回调；无实时条件时使用批量文件 | 公司出货不等于门店实际销售 |
| 智慧农场/养殖 | 存栏、日龄、健康、饲料、出栏和成本 | 业务系统接口或现有数据平台 | 与养殖计划、屠宰和生产批次关联 |
| 飞书 | 审批、任务、组织、文档、知识、日报和补充信息 | 飞书开放平台接口、消息回调 | 沟通内容需经核实后才能进入经营事实 |
| 外部市场 | 竞品价格、行业供需、政策、海关、检疫、汇率、物流和天气 | 官方或授权数据服务；公开网页只做线索 | 记录来源、时间、许可和可比性 |

采集速度不宜统一宣称为“实时”。具备消息回调或增量变更能力的系统可秒级接收；只有接口的系统按业务时效进行分钟级或小时级查询；只能提供文件的数据按约定批次处理。页面应同时显示业务发生时间、源系统记录时间和平台接收时间。

#### 运行过程

~~~text
源系统或外部来源
→ 身份校验与数据接收
→ 保留原始记录
→ 统一 SKU、门店、区域、价格、批次和时间口径
→ 检查完整性、时效性和数据冲突
→ 按企业批准的条件登记待调查信号
~~~

#### 责任与待核实事项

业务部门批准经营概念、指标口径和合法例外；数据和 IT 部门确认权威数据源、接口和主键；安全和法务部门确认外部数据许可及敏感字段权限。项目开始前需取得数据源清单、字段说明、接口日志、样本数据、法人范围和当前负责人。

### 5.2 第二组：调查与决策支持

| 模块 | 管理目的 | 主要产出 |
|---|---|---|
| M05 事件受理与分层调查 | 围绕真实业务问题开展调查，并随证据变化调整范围 | 原因假设、调查计划、证据请求、调查记录和当前判断 |
| M06 情景测算与决策材料 | 将事实、企业目标和业务约束转化为可比较的多种方案 | 备选方案、影响范围、假设条件、风险、证据缺口和审批草案 |

#### 为什么需要

传统报表告诉管理者“发生了什么”，这组能力继续回答“为什么发生、还缺什么证据、有哪些选择、每个选择的代价是什么”。智能助手适合做语义理解、原因归纳和调查路径选择；金额、毛利、库存、产能和现金流必须由可复算的模型或程序计算。

#### 运行过程

1. 识别业务对象、当前范围、时间、企业目标和不能突破的约束。
2. 列出相互区别的可能原因，同时保留反例和其他解释。
3. 每次只查询当前判断所必需且已授权的信息。
4. 系统查询与人工补充信息并行，进度在页面上公开显示。
5. 检查来源、时间、口径、冲突、权限和是否过期。
6. 决定停止、等待补充、转人工，或由单店扩大到区域、跨部门和集团层面。
7. 将事实、原因、方案、影响、风险、缺口和任务建议放入同一份决策材料。

每次调查都应设置查询次数、总时长、可用工具、证据要求和费用上限。等待人工补充信息时，当次调查先结束；收到新信息后根据事件编号继续，避免一次会话无限持续。

#### 责任与待核实事项

业务负责人批准调查方法、经营目标、方案选项和停止条件；财务和经营分析确认金额、毛利和成本口径；数据团队负责可复算模型和数据质量。试点前需取得一份真实决策材料、相应证据、管理者修改记录和实际处置结果，用于校准系统输出。

### 5.3 第三组：审批与责任执行

| 模块 | 管理目的 | 主要产出 |
|---|---|---|
| M07 管理审批与执行授权 | 确保重大经营决策由有权管理者批准，并将批准范围落实到系统权限 | 决策版本、审批结果、适用范围、预算、时间和执行授权 |
| M08 任务编排与事件协作 | 把批准的决策拆成可执行、可检查的跨部门任务 | 任务清单、负责人、配合人、前置关系、完成标准和事件协同群 |
| M09 岗位指导与案例应用 | 降低新员工和低频任务的学习成本 | 岗位职责、操作步骤、相似案例、注意事项和求助通道 |
| M10 业务进度与阶段验收 | 在正确业务节点检查结果，同时识别进度延误和执行质量问题 | 节点清单、源系统状态对应、合理处理时间、进度提醒和验收结果 |

#### 为什么需要

这组能力解决“谁做决定、谁执行、何时完成、用什么证明完成”。系统不以一段通知替代管理，而是把管理层批准的内容转化为可追踪的责任任务。

#### 运行过程

~~~text
确认审批有效
→ 根据业务模板生成任务草案
→ 确认负责人、配合人、前置任务和完成依据
→ 完成权限和法人范围校验
→ 下发任务，必要时建立事件协同群
→ 根据源系统状态跟踪业务进度
→ 到达规定节点后检查执行质量
→ 完成、延误升级、重新执行或重新审批
~~~

事件协同群的首条信息应包括事件背景、已批准内容、当前任务、负责人、截止时间、完成依据、权限限制和 Aily 追问入口。群聊是协作场所，审批、任务状态、价格和库存等正式事实仍以授权系统为准。

责任人接到任务后，可以选择“我是新入职人员”或直接提问。系统根据岗位、当前任务和权限，给出本岗位应负责的内容、操作步骤、所需证据、相似案例和求助对象。指导内容必须引用有效制度和已审核案例，不能由模型临时编造操作规程。

#### 责任与待核实事项

管理层批准决策；流程负责人维护任务模板和升级路径；岗位负责人执行并提交证据；专业复核岗位负责验收。项目开始前需取得一个真实跨部门决策样本、责任分工、审批层级、任务延误处理方式、飞书建群权限和业务节点样本。

### 5.4 第四组：风险预判与经营治理

| 模块 | 管理目的 | 主要产出 |
|---|---|---|
| M11 结果检查、风险预判与经营治理 | 判断执行是否真正改善经营结果，并将销售变化传导到库存、生产、养殖和市场决策 | 执行结果、风险预警、短中长期情景、存栏和产能建议、备选市场方案 |

任务完成不等于经营目标达成。例如门店已完成价签更换，还需检查消费者实付价是否正确，销量、库存和毛利是否达到预期。风险预判也不是给出唯一答案，而是在不同假设下测算企业可能面对的库存、毛利、现金流、产能和供应风险。

#### 运行过程

1. 批准方案时同时记录目标、基准、观察周期、停止条件和恢复办法。
2. 执行后按规定时间读取价格、销量、毛利、库存、回款、退货、投诉和任务证据。
3. 将试点结果与同口径基准、对照区域或历史季节性比较。
4. 结果改善时关闭事件并保留证据；无效或引发新风险时扩大调查、恢复原方案或重新审批。
5. 多区域持续出现动销放缓、降价和库存上升时，同时开展需求、库存、生产、存栏和外部市场分析。

| 需防范风险 | 主要表现 | 控制方式 |
|---|---|---|
| 指标口径变化 | 表面异常实际来自核算方法或并表范围变化 | 保留口径版本、生效日期和勾稽关系，不跨口径直接比较 |
| 渠道库存不透明 | 公司出货正常，门店实际销售放缓 | 同时读取公司出货、渠道库存和门店动销，无数据时明确标注 |
| 短期促销损害长期价格 | 为抢量扩大降价，引发渠道跟价和毛利下降 | 限定门店、数量和时间，设置预算及停止条件，检查价格外溢 |
| 预测过度自信 | 将单一模型结果当作确定未来 | 同时展示多种假设、置信范围、敏感性和未知信息 |
| 过早调整存栏或产能 | 一次短期波动触发长周期生产调整 | 使用多周期证据和业务约束，先采用可恢复措施，并由管理层审批 |
| 外部信息不可靠 | 将搜索结果或第三方页面当作政策和战略事实 | 重大判断回到官方或授权来源，并经法务、质量和业务确认 |

销售、供应链、生产、养殖和财务共同确认预测口径与业务约束；管理层批准存栏、产能、库存、价格和市场进入决策。实施前需取得各领域计划周期、最小调整幅度、硬性合同、产能上下限、库容、现金流要求和审批层级。

### 5.5 第五组：管理复盘与持续改进

| 模块 | 管理目的 | 主要产出 |
|---|---|---|
| M12 跨事件管理复盘 | 从多个同类事件中识别重复性管理短板，避免只处理个案 | 复盘批次、重复问题、反例、其他解释、改进选项和管理层报告 |
| M13 经营口径、判断条件和案例管理 | 保证经营知识随制度、组织和市场变化而更新 | 新版口径、判断条件、业务处置模板、模型、案例和发布记录 |

#### 为什么需要

企业管理不能长期依靠少数熟练员工记住例外和处理经验。已反复验证的口径、制度、判断方法和案例需要形成持续更新的经营知识。同时，十个事件出现相似现象也不能自动证明企业普遍存在某种问题。复盘必须同时提供证据、反例、其他解释和适用范围。

#### 运行过程

~~~text
同一经批准的业务领域
→ 筛选已关闭、结果已验证且证据完整的事件
→ 按根本问题去重，每十个事件形成一批
→ 比较背景、原因、处置过程和结果
→ 提出制度、流程、组织、系统、培训和风险管理方面的候选问题
→ 向管理层提交复盘报告
→ 管理层批准后形成改进任务
→ 用后续批次检查问题是否减少
~~~

第 1-10 个事件形成第一批，第 11-20 个形成第二批，不重复使用同一事件。上一批被驳回、暂缓或未发现稳定规律，不影响新批次继续形成。没有发现系统性问题时，报告应明确写出“本批次未发现稳定的重复性问题”，不为形式完整而强行归纳。

改进效果必须按企业批准的条件判断，包括最小的改进后事件数、订单或门店覆盖量、稳定观察时间和口径可比条件。未达到条件时，只能写“当前证据不足以判断改进效果”。

各业务领域负责人批准事件分类、证据完整标准和去重规则；管理层指定报告接收人和审议流程；内审、风险管理或专业部门参与高影响问题核对。试点需取得至少十个同领域脱敏历史事件，用人工复盘结果检查系统归纳是否准确。

## 6. 飞书与 Aily 的落地方式

### 6.1 飞书在方案中的定位

飞书承担人与系统协作的入口：显示事件、收集补充信息、提交审批、下发任务、建立协同群、查看进度和开展岗位问答。SAP、WMS、TMS、POS 等业务系统继续保存订单、库存、价格、生产和财务等正式事实。这样可以避免用飞书重做一套 ERP。

### 6.2 Aily 与后台服务的分工

| 组成 | 主要责任 |
|---|---|
| Aily 智能助手 | 理解问题、归纳原因、选择已授权查询、解释证据冲突、组织决策材料、回答岗位问题 |
| 后台可靠性服务 | 数据接入、口径转换、信号检测、事件状态、模型计算、权限校验、建群、任务创建、审计和失败重试 |
| 飞书页面与流程 | 对话、卡片、审批、任务、群聊、知识和看板 |
| 业务系统 | 保存正式业务事实，执行经授权的经营动作 |

本方案建议使用一个统一的智能助手运行环境。后台连接器、事件库、计算服务和权限服务是普通软件能力，不需要再建立一个与 Aily 竞争的外部智能助手。只有 Aily 无法满足模型、隔离、运行时间或工具调用要求时，才重新评估外部方案。

### 6.3 已核实的公开能力和仍需验证的内容

飞书开放平台已公开建群、群成员管理和发送消息等接口文档，说明自动建立事件协同群在平台层面具有实现基础。具体到圣农租户，还需确认应用是否开通，是否拥有建群、加人、发消息和读取组织信息的权限，以及接口频率、安全和数据保留要求。

飞书官方客户案例披露，圣农集团曾将多个系统审批集成到飞书统一待办，并使用多维表格、机器人和任务开展品质整改。这些材料证明相关实践曾经存在，不证明 2026 年的当前配置和覆盖范围。

### 6.4 两天内无法完成的技术核实

课题组没有圣农目标飞书租户、应用授权、内部接口凭证和真实业务样本，因此无法在两天内完成：

- SAP、POS、CRM、WMS、TMS 等系统的实际接口、字段、主键、延迟和错误处理验证；
- 圣农飞书租户内 Aily、审批、任务、建群、加人和机器人权限验证；
- 实际组织目录、法人范围、岗位权限、代理人和人员撤权后的同步验证；
- 从真实价格信号、跨系统调查、管理审批、任务下发到结果检查的完整流程测试；
- 接口断开、数据迟到、重复消息、权限不足、建群失败和模型输出错误等异常测试。

因此，本方案把技术路径写到“可以实施和验收”的程度，但不把“平台有公开能力”表述为“圣农已经配置完成”。正式实施时，应先用一个代表性业务流程完成端到端测试，再根据结果修正设计。

## 7. 实施路线与验收方式

### 7.1 先做一个可检查的价格事件

第一阶段选择一个区域、一个代表性渠道、一个 SKU、一条真实责任链和一次能够取得完整证据的价格事件。样本同时包含合法消费券、未取得审批证据的价格变化、规格差异、数据迟到、货物仍在运输或登记、管理层驳回或修改方案等情况。

只有这组样本能从信号来源追溯到最终结果，并证明合法例外不会被误判，才适合扩展到更多区域和业务领域。

### 7.2 分阶段实施建议

> 下列周期仅为讨论区间，真实周期取决于数据、接口、权限、采购、安全审查和业务人员投入。

| 阶段 | 建议周期 | 主要任务 | 进入下一阶段的条件 |
|---|---:|---|---|
| 0. 事实和范围确认 | 2-4 周 | 选定试点，还原流程，确认数据源、权限和负责人 | 关键信息均有负责人，无法取得的信息有替代办法 |
| 1. 价格信号试点 | 4-6 周 | 建立原始证据、判断条件、事件页面和人工补充信息流程 | 合法优惠、规格差异、未批准低价和数据失败样本能正确区分 |
| 2. 调查与审批试点 | 4-8 周 | 配置 Aily 调查、受控工具接口、决策材料和审批草案 | 权限、证据不足、跨天继续和模型错误等情况通过测试 |
| 3. 责任执行试点 | 4-8 周 | 配置任务、协同群、岗位指导、业务节点和结果检查 | 不过早告警；建群失败不影响任务；关闭有结果证据 |
| 4. 管理复盘试点 | 取得十个事件后 4-6 周 | 确认分类、去重和证据标准，开展人工与系统对照复盘 | 不重复计数、不强行归纳、不绕过管理层审批 |
| 5. 库存与产销风险 | 2-3 个月 | 扩展到多区域动销、库存、生产、存栏和产能情景 | 多种原因、业务约束和管理审批通过验证 |
| 6. 战略研究 | 持续完善 | 增加外部市场、政策、贸易、海外准入和长周期产能研究 | 来源、合规、模型、审批、试单和回款可追溯 |

### 7.3 验收指标

| 指标类别 | 核心指标 | 证明什么 |
|---|---|---|
| 经营结果 | 门店实际销售、库存周转、贡献毛利、回款、退货、投诉和复发 | 执行后是否真正改善经营 |
| 处理效率 | 发现、调查、补充信息、审批、执行和结果检查时间 | 瓶颈在数据、系统、审批还是业务执行 |
| 证据质量 | 完整度、来源可靠性、冲突率、过期率和可复算率 | 决策依据是否可信 |
| 调查质量 | 原因覆盖、无依据结论、工具成功率、证据不足判断准确率和人工修改率 | 智能助手是否按企业方法调查 |
| 检测质量 | 误报、漏报、合法例外识别、口径差异拦截和过早告警 | 程序检测是否可信 |
| 技术可靠性 | 传输延迟、重复、乱序、失败恢复时间、可用性和运行成本 | 系统是否适合正式运行 |
| 权限与安全 | 越权尝试、未审批执行、敏感数据暴露和审计覆盖 | 人工审批和权限控制是否有效 |
| 岗位成长 | 首次完成率、返工率、独立完成周期、主管辅导时间和同类问题复发 | 岗位指导是否缩短员工成才周期 |
| 管理复盘 | 合格事件比例、去重准确率、报告投递、改进批准和问题复发 | 是否把重复个案转为可审议的管理问题 |

### 7.4 失败处理和恢复

- 数据接口失败时标注“未取得”，不将缺失值当作零。
- 智能助手无法生成符合字段规范的结果时，保留证据并转人工处理。
- 飞书建群或添加成员失败时，正式任务继续下发，同时显示失败原因、重试进度和人工建群入口。
- 模型或外部信息不可用时，保留原有人工流程，不阻断 ERP 和专业系统的正常交易。
- 管理层批准的每个方案都应有停止条件和恢复办法，并保留受影响的事件、任务和数据记录。

## 8. 预期价值与管理层决策请求

### 8.1 预期价值

| 价值方向 | 衡量方法 |
|---|---|
| 缩短经营问题处理时间 | 分别测量发现、调查、补充信息、审批、执行和检查的基准时间与试点时间 |
| 减少价格、库存和履约损失 | 使用同口径异常暴露量、单位损失、返工和复发数据，排除季节、并表和自然恢复影响 |
| 提高库存和资金使用效率 | 比较同口径平均库存、库龄、资金占用、折价和跌价风险，不把库存转移给渠道当作改善 |
| 减少重复找数据和整理材料 | 记录真实减少的重复工时，扣除转移给其他岗位的工作 |
| 缩短新员工成才周期 | 比较首次完成率、独立完成周期、主管辅导时间和同类问题复发 |
| 补足管理短板 | 比较同一管理问题改进前后的复发率、处理周期、返工成本和覆盖量，证据不足时不强行归因 |

本方案不在没有企业基准的情况下填写投资回报数字。正确做法是试点前测量基准，试点期使用同口径数据，尽可能设置对照区域或历史季节性基准，最后扣除开发、数据、运维、模型和培训成本，由管理层决定是否扩大建设。

### 8.2 建议管理层批准的下一步

1. 确认“双向经营运行机制 + 统一数据与证据基础”作为总体方向。
2. 选择一个真实价格事件和一条完整责任链作为首个试点。
3. 指定业务、财务、IT、数据、安全、风险管理和飞书流程负责人。
4. 提供试点所需的脱敏数据、接口清单、权限清单、判断条件和真实处置样本。
5. 允许在隔离环境完成从价格信号、调查、审批、任务、建群、业务节点到结果检查的代表性端到端测试。
6. 首个单事件流程稳定后，提供同领域十个脱敏历史事件，由业务、管理层、风险管理或内审共同验证复盘方法。

### 8.3 方案成功的标志

~~~text
经营事实更可信
→ 问题更早暴露
→ 原因更快查清
→ 管理层看到不同选择及其代价
→ 批准的决策转化为明确责任任务
→ 任务在正确业务节点被检查
→ 新数据证明结果改善，或要求重新判断
→ 重复问题转化为可审议的管理改进课题
~~~

## 附录 B：智能助手的业务处置模板与受控工具

### B.1 业务处置模板应按职责分层

| 层级 | 业务处置模板 | 触发 | 主要输出 |
|---|---|---|---|
| 统一受理 | `event-intake-loop` | 任意经营信号 | 已知/未知、原因假设、下一项最小查询、停止/扩大/等待/转人工 |
| 销售价格 | `price-anomaly-investigation` | 价格候选信号 | 价盘、规格、券、审批、承担方、动销证据链 |
| 竞品响应 | `competitive-response-pricing` | 对手让利/抢量 | 跟价、定向券、规格区隔、不跟价情景 |
| 库存风险 | `inventory-demand-risk` | 多区降价、滞销、库龄或减值信号 | 需求、渠道库存、生产、存栏和替代市场假设 |
| 战略研究 | `strategic-market-investigation` | 管理者主动问题 | 内外部来源计划、市场初筛、多情景和缺口 |
| 决策汇报 | `decision-briefing` | 证据达到决策就绪 | 事实、推断、缺口、方案、影响、风险、审批草案 |
| 批准后执行 | `approved-decision-execution` | 有效 approval_id | 跨部门任务草案、协同群范围草案、里程碑和回滚要求 |
| 结果验证 | `milestone-outcome-verification` | 里程碑/回检事件 | 进度风险、质量异常、恢复、升级、重开 |
| 管理复盘 | `management-review` | 同领域十个合格关单事件 | 零个或多个六维候选、反例、替代解释、历史关联、改进选项和验证条件 |
| 岗位辅导 | `role-guided-task-assistance` | 责任人在任务内追问 | 岗位职责、步骤、案例、权限和升级路径 |

统一受理 业务处置模板 定义所有事件共同的调查循环；领域 业务处置模板 定义企业老手面对某类问题时的专业检查方法。实时金额、库存和状态不写进 业务处置模板，而由工具查询；权限和动作限制由服务端执行，不能只写在 指令模板 中。

### B.2 五类受控工具接口

| 工具域 | 只读工具示例 | 受控写工具示例 | 核心权限与风险控制 |
|---|---|---|---|
| `commercial-facts` | `resolve_entities`、`get_price_policy`、`get_promotion`、`get_sellout`、`get_margin` | 原则上首期无正式写入 | 同款可比、字段级权限、口径版本 |
| `inventory-supply` | `get_inventory`、`get_lot_age`、`get_capacity`、`get_farm_plan`、`get_supply_plan` | `submit_plan_change_request` | 只生成申请；存栏/产能须审批 |
| `market-intelligence` | `get_competitor_price`、`get_market_signal`、`get_source_snapshot`、`check_comparability` | 无外部自动发布 | 来源许可、快照、时间和可比性 |
| `decision-model` | `calculate_impact`、`simulate_price`、`forecast_demand`、`simulate_inventory`、`compare_scenarios` | 无交易写入 | model_run_id、输入版本、可复算和适用范围 |
| `event-execution` | `get_event_context`、`get_task_state`、`get_approval`、`get_collaboration_state` | `append_evidence`、`create_approval_draft`、`create_approved_tasks`、`create_collaboration_chat`、`sync_collaboration_members`、`archive_collaboration_chat`、`schedule_recheck` | 写操作幂等；建群按当前 RACI/权限二次校验；正式经营写入需 approval_id |

不要把每个 SAP/POS 接口直接暴露给模型。领域网关应裁剪动作空间、统一 数据字段规范、隐藏凭据并记录审计。

智能助手 只能提出历史关联，受权管理 owner 确认后才能合并管理问题 ID。`candidate_issues` 可以为空，此时 `systemic_issue_found` 必须为 `false`；每个候选的 `dimensions` 至少包含 `policy|process|organization|system|training|risk_control` 中一个维度。

管理问题 `status` 枚举为：`candidate|under_review|approved_for_pilot|approved_for_change|rejected|deferred|executing|observing|improved|unchanged|worsened|inconclusive|invalidated`。未取得管理审批时不能进入 `executing`；没有达到效果判断条件时只能进入 `inconclusive`。所有运行、证据、工具、模型、指令模板、业务处置模板、法人和权限标签随报告版本保存。

投递 `status` 枚举为 `queued|delivered|failed|read|acknowledged`。角色快照只用于审计；每次投递、重试和打开报告仍须根据当前组织目录重新鉴权。失败原因、重试次数、发送/已读/确认时间和审计 ID 必须可查询，领域负责人未响应不能阻断授权管理层接收。

## 附录 F：管理学大白话术语表

| 术语 | 大白话解释 |
|---|---|
| ERP/SAP | 企业正式记账和做业务的主系统，不等于 智能助手，也不等于一条聊天计划 |
| MOM/MES | 管生产现场、工序、质量和进度的运营执行系统，不只是把 ERP 数据做成好看的页面 |
| 语义层 | 告诉系统“企业里的词到底是什么意思、怎么计算、何时例外、谁有权处理” |
| 暗知识 | 人一看就懂、表里却没写明的区域范围、口径、规则、责任和边界 |
| 规则 | 企业批准的一段确定性判断逻辑；输入相同就应得到相同结果 |
| 信号包 | 一个值得调查的小信息点，尚未说明原因和责任 |
| 经营事件 | 围绕一个问题持续积累背景、证据、方案、责任和状态的对象 |
| 决策就绪事件包 | 管理层已经看得清事实、缺口、选项和代价，可以作出有边界决定的材料 |
| 管理复盘批次 | 同一领域十个已关闭、结果已验证且证据完整的不同问题组成的一组，不与其他批次重复 |
| 管理问题上报包 | 智能助手 比较十案后形成的候选管理短板材料，包含证据、反例、其他解释和改进选项，不等于既成事实 |
| 事件协同群 | 责任到人后由受控服务自动建立的最小成员飞书群，用来共享事件上下文和岗位追问；不是 ERP、审批或正式状态源 |
| 协作范围 | Aily 根据任务图提出、服务端按 RACI/法人/权限校验的群成员集合 |
| collaboration_idempotency_key | 事件、流程和协作范围的唯一组合，保证重复触发不会重复建群 |
| canonical_case_id | 给同一个根问题的统一编号，避免多条告警、任务或系统记录被重复算成多个问题 |
| effectiveness_policy | 企业批准的改进效果判断规则，规定最小干预后样本、暴露量、观察窗和可比条件 |
| 暴露分母 | 订单量、门店数、销量、巡检量等“有多少机会发生问题”的基数；没有它不能只凭十案断言总体风险下降 |
| inconclusive | 证据不足以判断有效或无效；不是成功，也不是失败，更不能由 智能助手 猜结论 |
| 业务处置模板 | 企业老手解决某类问题的方法包，不是实时数据库，也不拥有审批权 |
| 受控工具接口/工具 | 智能助手 受控查询业务系统或调用模型服务的接口 |
| API / Webhook | API 是系统按约定查询或写入数据的接口；Webhook 是有新事件时由源系统主动通知 |
| CDC | 只复制数据库发生变化的部分，适合经授权的高数据量增量同步 |
| SFTP / EDI | 通过安全文件或企业标准报文交换数据，速度可能较慢，但容易审计 |
| AnyCross | 飞书集成平台，用于连接系统和编排数据传输，不等于语义层、事件库或 智能助手 |
| sell-in / sell-out | 前者是圣农向渠道发货，后者是渠道真正卖给终端；只看前者可能只是移库 |
| SLA | 企业批准的处理时限和服务标准，例如多久认领、补证、审批或恢复 |
| 端到端测试 | 用一条代表性真实流程，从输入一直跑到结果和失败处理，证明各环节能够稳定衔接 |
| 数据字段规范 / 数据合同 | 一份数据必须有哪些字段、类型、版本和质量规则的共同约定 |
| RACI / 责任矩阵 | 谁最终负责、谁执行、需咨询谁、需通知谁的职责表 |
| owner / 负责人 | 对一项数据、规则、流程或结果长期负责并有权批准变更的人 |
| 幂等 | 同一事件重复到达，也只产生一次有效业务结果 |
| stale / 数据过期 | 数据曾经有效，但超过允许更新时间，不能继续当作当前事实 |
| hash / 数字指纹 | 用于证明文件或记录是否被改变的一串校验值 |
| 里程碑 | 业务真正走到哪个节点的正式证据，不是员工口头说“做完了” |
| 阶段验收检查 | 里程碑到达并过宽限期后，才运行的检查 |
| 回滚/补偿 | 决策失败后怎样停止、撤销可逆动作并修复不可逆影响 |
