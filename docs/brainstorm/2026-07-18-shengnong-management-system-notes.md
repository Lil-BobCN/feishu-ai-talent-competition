# 圣农经营管理系统 Brainstorm 记录

状态：持续讨论中，非最终方案，未经用户确认不得作为正式架构或实施承诺。

## 1. 当前目标

围绕圣农命题，先充分扩展系统应该解决的问题和应具备的能力，再统一收敛为方案框架。当前重点不是立即实现，而是把分散在任务、聊天、研究材料中的判断沉淀为可验证的问题链。

## 2. 已确认的基础判断

1. 语义层是底座：统一商品、价格、门店、渠道、区域、业务员、库存、促销、产能、养殖与市场等对象的名称、口径、关系和证据。
2. 异常不是简单的数据差异：必须同时包含预期基准、实际状态、偏差阈值、合法例外、证据状态和责任归属。
3. 规则和分析负责确定性判断；Agent 负责跨系统取证、解释、协调、方案生成、任务拆解和持续验证。
4. 飞书承接一线采集、展示、问数、任务、审批和组织协作，但不是替代 ERP、数据仓库或正式交易系统。
5. 高风险决策必须由管理层批准。存栏、产能、价格、市场进入等事项不得由系统自动拍板。
6. 管理层批准后，系统应把决策拆成养殖、生产、库存、销售和海外业务任务，指定负责人、时限和验证指标，并持续跟踪结果。

## 3. 风险预判模块的已确认方向

系统不能只回答“现在发生了什么”，还应回答：

- 当前异常可能演变成什么经营风险；
- 风险会如何沿销售、渠道、库存、生产、养殖和市场链条传导；
- 不采取行动时可能产生什么后果；
- 可以采取哪些规避方案；
- 每个方案的收益、成本、时效、约束和副作用是什么；
- 管理层批准后，如何形成跨部门执行闭环；
- 执行后用什么指标验证风险是否下降。

尚未对齐：风险预测应覆盖的时间尺度，以及不同时间尺度对应的决策权和数据粒度。

## 4. 贯穿讨论的真实案例

候选主案例：产品价格压力与库存价值风险。

圣农发展 2025 年年度报告披露：

- 公司面对产品价格压力与市场挑战；
- 2025 年末存货账面价值约 32.86 亿元；
- 当年对库存商品等存货计提资产减值约 1.98 亿元；
- 公司白羽鸡养殖产能近 8 亿羽，食品深加工产能超过 50 万吨；
- 公司提出从育种、饲料、养殖、屠宰、物流到终端销售实施全周期动态监控与决策优化。

证据边界：以上事实不能直接证明全国白羽鸡需求下降。可能原因还包括供给扩张、并购并表、渠道变化、产品组合、促销竞争、季节波动、库存结构和数据口径变化。系统必须先形成多种解释并验证，不得从“降价或库存上升”直接跳到“降低存栏”。

## 5. 当前深化主题：同一数据的最大化复用

原始问题：一线和部门已经在使用销售、价格、库存、铺货、促销、客户和现场数据，这些数据如何继续服务集团高管的战略决策？

初步抽象：同一份数据的价值不在于被多个 Agent 重复读取，而在于经过不同层级的语义组合，回答不同管理问题。

### 一线执行层

- 我今天应该去哪些门店；
- 哪个商品缺货、低价或没有陈列；
- 哪项任务逾期；
- 当前客户政策和促销权限是什么。

### 区域或部门管理层

- 哪类异常反复发生；
- 哪些门店、经销商、商品或业务员贡献了主要偏差；
- 区域资源应如何调整；
- 任务执行是否真正改善销量、价格、库存和毛利。

### 集团经营决策层

- 多个区域同时出现的现象是局部执行问题，还是市场结构变化；
- 销量、价格、毛利、库存和竞品动作之间出现了什么组合信号；
- 哪些产品、渠道和区域正在创造增长，哪些正在消耗现金与产能；
- 生产、库存、养殖、产品组合和市场布局需要怎样调整；
- 不同策略在未来各时间尺度下的结果和风险是什么。

### 战略层数据复用的核心变化

```text
一线数据点
-> 标准经营事件
-> 跨门店/区域/渠道聚合
-> 时间趋势和结构变化
-> 风险假设与原因验证
-> 多情景方案比较
-> 管理层审批
-> 跨部门任务
-> 结果验证与模型校准
```

战略层不应只得到更大的可视化看板。它需要的是把分散数据转化为“需要管理层决定什么、为什么现在决定、有哪些选择、每个选择会怎样”的决策材料。

## 6. 战略目标冲突：系统到底应该优化什么

销量、利润、库存周转、现金流和市场份额不是天然一致的指标。系统如果没有管理层明确的目标顺序，同一批数据可能导出互相矛盾的建议。

### 五类目标分别代表什么

- 销量：商品卖出多少，能反映市场接受度和渠道消化能力，但降价也可能制造短期销量。
- 利润：收入扣除成本后的经营回报，但只看当期利润可能牺牲市场、客户关系和未来增长。
- 库存周转：库存转化为销售的速度，过慢会占用资金并增加减值风险，过快也可能导致缺货。
- 现金流：企业是否有足够现金维持采购、养殖、生产、工资和偿债，是持续经营的底线。
- 市场份额：企业在行业和渠道中的竞争位置，但以持续低价换取份额可能破坏利润和价格体系。

### 同一风险下的决策冲突

假设多个区域同时出现销量下降、门店降价和库存上升：

1. 继续降价促销：可能提高销量、改善库存周转和回笼现金，但会降低单品毛利，并可能破坏长期价格体系。
2. 坚持价格：可能守住毛利和品牌定位，但销量、库存和现金占用风险可能继续恶化。
3. 降低生产和存栏：可以减少未来供给压力，但可能降低产能利用率、抬高单位成本，并存在需求恢复时无法及时供货的风险。
4. 转做深加工或调整产品组合：可以延长保存周期、提升附加值，但需要加工产能、研发、包装和渠道资源。
5. 开拓海外市场：可以分散国内需求风险，但涉及认证、客户开发、贸易、物流、汇率和回款周期，不能作为即时去库存动作。

因此，系统不能生成一个脱离企业目标的“最优答案”。它应先识别当前经营阶段和管理约束，再给出多情景方案。

### 已确认的目标层级

用户确认：圣农作为白羽肉鸡全产业链企业，首要经营目标必须是保销量和保证库存周转维持在正常水平。养殖端的种蛋、雏鸡、肉鸡均属于生物资产，出栏前的饲料、人工和间接费用会持续进入成本；市场消化不及时会沿整条产业链形成成本、库存和减值压力。

1. 硬约束：食品安全、生物安全、合规、核心客户履约和重大风险容忍度，任何方案都不得突破。
2. 第一经营目标：保持有效销量，确保终端真实需求能够持续消化产能。
3. 第二经营目标：保持从养殖、屠宰、加工、公司库存、经销商到门店的全链条库存周转处于健康区间。
4. 经营约束：毛利、现金流、应收账款、价格体系、渠道库存和产能利用率不得因虚假保销量而失控。
5. 战略目标：在销量与周转健康的基础上，提高市场份额、调整产品结构、发展新品和开拓海外市场。
6. 时间边界：区分短期去库存、中期产销平衡和长期养殖产能/市场布局，不能用同一个权重处理所有时期。

### “保销量”必须防止口径造假

保销量不能只看圣农向经销商发货的 `sell-in`。如果货物只是从圣农仓库转移到经销商仓库，集团报表上的销量提高了，但终端需求没有增加，库存风险只是被转移到渠道。

系统需要同时观察：

- `sell-in`：圣农向经销商或客户发货；
- `sell-out`：门店或渠道向最终消费者售出；
- 渠道库存：经销商和门店尚未消化的库存；
- 回款：销售是否已经转化为现金或可靠应收；
- 退货与折价：销量是否依赖未来退货、返利或持续降价。

第一经营目标更准确地说是“保有效销量”：以终端动销和真实回款为核心，在不破坏价格体系和渠道健康的前提下维持市场消化能力。

用户已确认：以消费者真实购买的终端动销 `sell-out` 作为保销量的首要验收口径，而不是以圣农向经销商完成发货作为核心口径。

### “库存周转”必须分层计算

圣农的库存不是一个仓库里的单一商品，而是处在不同生命和加工阶段的多层资产：

```text
种蛋/雏鸡/肉鸡
-> 待屠宰与鸡肉产品
-> 深加工产品
-> 公司仓库
-> 经销商库存
-> 门店库存
-> 消费者购买
```

各层的时间窗口、成本累积、保质期和处置方式不同。系统必须分别设置正常周转区间，并识别库存是否只是从上游转移到下游，而没有被市场真正消化。

管理层可以在不同经营阶段调整具体阈值，但每次调整都应记录批准人、生效时间、适用业务和风险上限。

### 已确认的双价格底线

用户确认：

1. 正常经营期必须守住公司价格体系和贡献毛利底线，不能为保销量无限降价。
2. 当临期库存、活体生物资产积压等风险继续持有会造成更大损失时，可以比较继续持有损失与降价销售损失。
3. 特殊处置必须由管理层审批，按损失最小化原则执行；Agent只能提供测算、证据与方案，不能自行突破价格底线。
4. 特殊处置不能常态化，必须限定商品、区域、渠道、数量、时间和退出条件，并监控对正常价格体系的影响。

### 已确认的三层风险预测时间尺度

用户确认风险预判同时覆盖三个时间尺度，并共享同一语义数据底座：

1. 短期（未来 1-30 天）：终端动销、价格、促销、门店和渠道库存，服务保销量、快速去库存和价格异常处置。
2. 中期（未来 1-6 个月）：订单、生产、屠宰、加工、仓储和渠道需求，服务生产计划、产品组合与区域资源调整。
3. 长期（未来 6-18 个月）：种鸡、鸡苗、养殖产能、行业供需和海外市场，服务存栏、产能投资、市场布局与战略客户决策。

三个尺度使用不同的数据粒度、预测模型、置信度要求、审批层级和停止条件，不能用一个通用预测结果替代。

### 对系统能力的直接影响

系统需要从“单指标预测”升级为“多目标、带约束的情景比较”：

```text
经营目标与硬约束
-> 当前事实与风险信号
-> 多种行动方案
-> 分别预测销量/利润/库存/现金流/份额变化
-> 显示收益、代价、置信度和不可逆风险
-> 管理层审批
-> 执行与结果验证
```

Agent负责解释目标冲突、组织证据和生成决策材料；预测模型负责估算不同情景；规则引擎负责阻止违反硬约束的方案；管理层保留最终取舍权。

## 7. 待继续对齐的问题

当前最关键的问题：战略风险判断是否允许并要求接入外部市场数据。仅靠圣农内部数据只能说明公司发生了什么，无法可靠区分内部执行问题与行业需求、竞争、原料、政策或国际市场变化。

## 8. 责任到人后的岗位辅导与成长闭环

用户新增确认：责任到人不能只完成任务派发，还必须缩短业务人员成才周期。负责人应在任务上下文中向 Agent 追问，获得与其岗位、区域、资历、当前里程碑和权限相匹配的指导。

### 8.1 两条并行线

```text
任务责任线：接收任务 -> 执行动作 -> 回写证据 -> 里程碑验证
岗位成长线：理解背景 -> 追问标准 -> 查看类似案例 -> 获得下一步指导
```

两条线共享同一个事件包和任务上下文，但责任边界不同：Agent 可以解释、提醒、检索制度和历史案例，不能代替负责人确认现场事实、提交业务承诺或越过审批修改正式数据。

### 8.2 任务下发内容

责任人收到的任务必须包含：

- 当前经营事件和背景；
- 触发证据、当前里程碑和完成标准；
- 本岗位需要执行的动作；
- 依赖任务、权限边界和升级联系人；
- 截止时间、后置验证节点和需要回写的证据。

### 8.3 任务内追问能力

负责人可以直接在任务上下文中询问：

- “我作为新入职人员，第一步应该做什么？”
- “这个岗位的完成标准是什么？”
- “过去类似问题是怎么处理的？”
- “我缺少什么信息或权限，应向谁申请？”

统一事件受理 Skill 负责说明当前问题和缺失信息；领域 Skill 负责提供岗位化操作步骤、制度依据和历史案例。回答必须带来源、适用范围和版本，不能只返回无依据的经验性建议。

### 8.4 技术能力

需要新增或扩展：

1. **岗位与责任目录**：岗位、部门、区域、资历、权限、数据责任域和升级链路；
2. **任务上下文服务**：把事件包、里程碑、任务、制度和相关历史案例绑定到同一会话；
3. **岗位化 Skill 路由**：依据岗位、任务类型、调查层级和当前节点选择指导内容；
4. **案例与复盘库**：保存已验证的处理过程、适用条件、结果和失败原因；
5. **学习结果指标**：追踪追问类型、首次完成率、返工率、异常复发率和独立完成周期。

### 8.5 可见性和护栏

任务页面同时展示：

```text
任务状态 + 里程碑状态 + 证据状态 + 指导记录 + 审批状态
```

Agent 的指导记录、负责人提交的事实、审批意见和最终结果都要留痕。指导不改变责任归属；高风险价格、存栏、产能和市场进入决策仍必须由管理层审批。
