# Aily 事件协同群设计规格

> 日期：2026-07-19
>
> 状态：用户已明确要求，按安全默认边界实施
>
> 适用文档：`docs/圣农经营智能中枢_完整方案母稿.md`
>
> 能力性质：方案设计；飞书平台通用 API 已公开，圣农目标租户权限、套餐和代表性 E2E 仍待验证。

## 1. 目标

当经营事件进入“责任到人”阶段，Aily 不只生成任务，还能触发飞书自动建立该事件的协同群，把当前阶段真正需要协作的负责人纳入同一上下文。群内通过固定事件卡片同步背景、证据、批准方案、任务、里程碑、缺口和截止时间，并允许责任人向 Aily 追问岗位动作和历史案例。

这项能力解决“任务已经分配，但跨部门人员仍要人工找人、拉群、重复解释背景”的协作损耗。群聊不是第二套业务系统，也不替代 SAP/WMS/TMS、正式审批、任务状态和事件库。

## 2. 平台事实边界

飞书开放平台已经公开以下服务端能力：

- [创建群聊](https://open.feishu.cn/document/server-docs/im-v1/chat/create)；
- [将用户或机器人加入群聊](https://open.feishu.cn/document/server-docs/im-v1/chat-members/create)；
- [发送消息](https://open.feishu.cn/document/server-docs/im-v1/message/create)。

这些公开页面证明飞书提供通用 API，不证明圣农租户已经给本方案应用开通相应权限，也不证明 Aily 可以无需配置直接调用。目标实现应由 Aily Workflow/Skill 通过自建 MCP 或受控工具调用事件协同群服务，再由该服务持有飞书应用凭据和执行 API。

## 3. 推荐架构

```text
M07 有效审批 / M08 任务图已校验
→ Aily 根据事件和任务图提出协作范围
→ 事件协同群服务重新读取 RACI、组织、法人和当前权限
→ 计算最小成员集合
→ 幂等创建群聊并添加成员/机器人
→ 发送并置顶事件上下文卡片
→ 任务、审批、里程碑和补证状态持续同步
→ 事件关闭并过观察期后归档；重开时继续同一协作上下文或建立可追溯替代群
```

Aily 不直接持有飞书凭据，也不能仅凭提示词自行决定拉入谁。确定性服务负责身份解析、权限、幂等、成员变更、重试和审计。

## 4. 触发规则

1. 默认触发点是 M08 已根据有效 `approval_id` 生成并校验任务图、负责人和依赖之后。
2. 调查阶段若只需要某个人补证，继续使用补证卡，不为每次查询都建群。
3. 当任务涉及两个以上主动责任角色，或跨部门/跨法人协作时，自动建群；单人任务默认只发任务卡，避免群聊泛滥。
4. 重大事件可以由有权限的事件负责人手工提前触发调查协同群，但这不授予任何未经批准的经营动作权限。
5. 幂等键使用 `event_id + process_id + collaboration_scope_version`；同一有效协作范围只能有一个活动群。

## 5. 成员规则

默认最小成员集合包括：

- 事件负责人；
- 当前任务的直接责任人；
- 任务图中必须协同的负责人；
- 当前缺失证据的责任人；
- 事件升级负责人或指定管理者；
- Aily/事件机器人。

仅被知会人员默认不自动拉入，可通过事件看板查看脱敏状态。外部经销商、客户、供应商和个人账号不得自动加入内部事件群；确需外部协同时，必须使用经过批准的外部群模板和数据边界。

成员在建群、重试加入和打开敏感事件卡片时都要重新校验当前组织、法人和字段权限。转岗、离职、代理或撤权后，服务根据组织事件更新成员；历史快照只用于审计，不能替代实时鉴权。

## 6. 群内首条事件卡片

群名使用不暴露敏感客户/财务信息的模板，例如：

```text
[经营事件] PRICE-0249｜华东区域价格处置
```

首条卡片至少包含：

- `event_id`、`process_id`、`decision_version` 和群聊 ID；
- 经权限裁剪后的事件背景、已知事实和证据链接；
- 管理层批准内容、适用范围、禁止事项、停止/回滚条件；
- 责任人、任务、依赖、当前里程碑、SLA 和完成证据；
- 当前缺口、补证责任人和升级路径；
- “向 Aily 追问本岗位下一步、完成标准或类似案例”的入口；
- 系统记录边界：群聊讨论不等于审批、任务完成或源系统业务状态。

## 7. Aily 在群内能做什么

- 按成员岗位和当前事件上下文解释“为什么找我、我现在做什么、怎样算完成”；
- 引用现行制度、SOP 和已验证案例；
- 汇总任务、补证和里程碑状态，提醒即将逾期的责任人；
- 把群内新事实转成待核验证据，写回事件服务前保留来源和提交人；
- 根据已批准任务图提出缺失协作角色，由服务校验后再加人。

Aily 不能把聊天共识当正式审批，不能自动扩大成员的数据权限，不能在群内直接决定价格、存栏、产能、市场进入、制度或组织变更。

## 8. 生命周期与失败处理

| 场景 | 系统行为 |
|---|---|
| 重复触发/消息重放 | 返回现有群聊 ID，不重复建群 |
| 部分成员加入失败 | 任务照常下发；卡片标出失败成员、原因和重试状态；通知事件负责人 |
| 建群 API 失败 | 进入 `collaboration_degraded`；继续个人任务/卡片；提供人工建群按钮和可重试名单 |
| 成员撤权或离岗 | 重新解析角色并移除/替换；保留成员变更审计 |
| 任务图或审批版本变化 | 更新群卡片和成员范围；旧版本明确失效，不静默覆盖 |
| 事件关闭 | 发送结果与关闭卡；过企业保留期后按策略归档，不删除正式证据索引 |
| 事件重开 | 优先恢复原群上下文；无法恢复时创建替代群并双向记录旧/新群 ID |
| 群消息含新事实 | 标为“待核验群内证据”，经权限、来源和业务校验后才能进入事件事实层 |

## 9. 数据合同最小扩展

任务/事件合同新增：

```json
{
  "collaboration_required": true,
  "collaboration_scope_version": "v1",
  "collaboration_chat_id": "oc_example",
  "collaboration_idempotency_key": "EVENT-001:PROCESS-001:v1",
  "collaboration_status": "pending|active|degraded|closing|archived",
  "participant_role_ids": [],
  "membership_snapshot_version": "org-v1",
  "last_membership_sync_at": "2026-07-19T17:30:00+08:00",
  "replacement_chat_id": null
}
```

所有建群、加人、移人、发卡、失败、重试和归档动作记录操作者、当前权限版本、飞书请求 ID、时间和结果。

## 10. 验收标准

1. 一个跨部门批准任务图只生成一个活动群，重复触发不重复建群。
2. 群成员与当前 RACI/补证/升级责任一致；纯知会人员和未批准外部人员不自动加入。
3. 撤权、离职、跨法人无权人员无法被加入或读取敏感卡片。
4. 首条卡片包含事件、批准、任务、里程碑、缺口、停止条件和岗位追问入口。
5. 群内 Aily 回答受当前成员权限约束，并引用制度/案例来源。
6. 没有 `approval_id` 时，群聊不能触发正式高风险经营动作。
7. 建群或加人失败不会把任务标为完成，也不阻断 SAP 等正式系统；降级、重试和人工入口可见。
8. 任务/审批/里程碑版本变化能同步到群卡片，旧版本不会继续被当作有效指令。
9. 事件关闭和重开均保留群聊与事件的双向追溯。
10. 在目标飞书租户完成创建群、加/移成员、机器人发卡、重复触发、撤权、失败重试和归档代表性 E2E 后，能力状态才可从 `需要 E2E` 升级。

## 11. 母稿更新位置

- 一页结论和共同尾段：责任任务之后增加事件协同群；
- 价格与产销存案例：展示批准后自动拉群、首卡和岗位追问；
- M08：新增事件协同群目的、触发、成员、生命周期和失败降级；
- M09：说明员工可在事件群中向 Aily 获取岗位辅导；
- 飞书/Aily 能力矩阵：加入创建群聊、群成员管理和机器人消息能力，标明目标租户 E2E；
- E2E 清单：加入幂等建群、成员权限、撤权、失败降级、关闭/重开；
- 任务合同：加入协同群字段；
- Skill/工具：批准后执行 Skill 输出协作范围草案，事件执行工具增加受控建群/成员同步；
- 术语表和题面回应：解释事件协同群不是第二套 ERP，也不是正式状态源。
